## 1. Architecture & OOAD (객체지향 분석 및 설계)
* **엄격한 계층 분리:** Controller(API 엔드포인트), Service(비즈니스 로직), Repository(데이터 액세스), Entity(도메인 모델) 간의 책임을 명확히 분리한다. 계층 간 의존성 침범을 엄격히 금지한다.
* **SOLID 원칙 준수:** 단일 책임 원칙(SRP)과 개방-폐쇄 원칙(OCP)을 최우선으로 적용한다. 비즈니스 로직 확장 시 기존 코드를 수정하기보다 인터페이스와 다형성을 활용하여 확장하도록 설계한다.
* **의존성 주입(DI):** 구현체가 아닌 추상화된 인터페이스에 의존하도록 설계하여 컴포넌트 간의 결합도를 낮춘다.

## 2. Database & Data Optimization (데이터베이스 최적화)
* **N+1 문제 원천 차단:** JPA/Hibernate 등 ORM 사용 시 Fetch Join 또는 EntityGraph를 활용하여 불필요한 추가 쿼리 발생을 방지하고 조회 성능을 최적화한다.
* **트랜잭션 관리:** 데이터 상태를 변경하는 모든 비즈니스 로직에는 `@Transactional`을 명시하고, 읽기 전용 작업에는 `readOnly = true` 속성을 적용하여 DB 락(Lock) 경합을 줄이고 성능을 극대화한다.
* **도메인 모델 캡슐화:** 엔티티의 무분별한 Setter 사용을 지양하고, 의미 있는 비즈니스 메서드(예: `changePassword()`, `updateProfile()`)를 통해 객체 내부 상태를 변경한다.

## 3. Data Structures & Algorithm (자료구조 및 성능)
* **적절한 컬렉션 선택:** 데이터의 검색, 삽입, 삭제 빈도를 논리적으로 분석하여 가장 효율적인 자료구조(List, Set, Map 등)를 선택한다. 모든 로직은 시간 복잡도(Time Complexity)와 공간 복잡도를 우선적으로 고려하여 작성한다.
* **대용량 데이터 처리:** 메모리 누수를 방지하기 위해 대규모 데이터 조회 시 페이징(Paging) 처리나 스트림(Stream) API를 필수적으로 적용한다.

## 4. AI Agent Behavioral Constraints (에이전트 행동 제어)
* **최소 권한의 원칙:** 지시받은 특정 도메인 로직 외에 다른 도메인 파일이나 전역 환경 설정 파일(`application.yml`, `build.gradle`, `pom.xml` 등)은 절대 임의로 수정하지 않는다.
* **사전 추론 강제:** 코드를 작성하거나 리팩토링하기 전에, 변경될 아키텍처 구조와 그로 인해 얻을 수 있는 성능적 이점을 먼저 마크다운으로 요약하여 사용자에게 보고한다.
* **테스트 주도:** 핵심 비즈니스 로직 수정 시, 발생 가능한 엣지 케이스(Edge Case)를 포함한 단위 테스트(Unit Test) 코드를 반드시 함께 작성하여 신뢰성을 검증한다.
